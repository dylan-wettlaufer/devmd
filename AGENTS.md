# Repository Guidelines

## Project Structure & Module Organization

This repository currently contains the product definition for DevMD in `PRD.md`. Treat it as the source of truth for product scope, MVP requirements, and future architecture decisions.

As implementation begins, use a conventional Next.js layout:

- `src/app/` for routes, pages, layouts, and server actions.
- `src/components/` for reusable UI components.
- `src/lib/` for GitHub parsing, LLM orchestration, Markdown rendering, and shared utilities.
- `src/types/` for shared TypeScript types and generation schemas.
- `tests/` or colocated `*.test.ts(x)` files for automated tests.
- `public/` for static assets.

Keep generated output examples, prompts, and schema fixtures in a clearly named folder such as `fixtures/` or `src/lib/generation/fixtures/`.

## Build, Test, and Development Commands

No package scripts exist yet. After the app is scaffolded, document the actual commands here and keep them in sync with `package.json`.

Expected examples:

- `npm run dev` starts the local development server.
- `npm run build` creates a production build.
- `npm run lint` checks formatting and code quality.
- `npm test` runs automated tests.

Do not add command references unless they are available in the repository.

## Coding Style & Naming Conventions

Use TypeScript for application code. Prefer small, focused modules with descriptive names such as `analyze-repo.ts`, `render-project-brain.ts`, and `github-file-filter.ts`.

Use PascalCase for React components, camelCase for functions and variables, and kebab-case for non-component file names. Keep Markdown output structures aligned with the strict section list in `PRD.md`.

When adding formatting or linting tools, prefer the defaults generated by the selected framework and avoid broad style churn unrelated to the feature being changed.

## UI Components & Styling

When building UI, use shadcn components whenever an appropriate component is available. Check and install components through the connected shadcn MCP server so usage matches the configured registry and local project setup.

Always follow the provided color scheme for application styling. Do not introduce new brand colors, gradients, or palette changes unless the design requirements explicitly call for them.

## Testing Guidelines

Add tests around repo parsing, file filtering, prompt/schema generation, Markdown rendering, and regeneration behavior. Use clear test names that describe observable behavior, for example `filters build artifacts from repo tree`.

For UI work, include tests or screenshots for key states: URL input, analyzing/loading, generated result, editing, and section regeneration.

## Commit & Pull Request Guidelines

This directory does not currently expose Git history, so no existing commit convention can be inferred. Use concise, imperative commit messages, for example `Add repo analysis schema` or `Implement markdown export`.

Pull requests should include a short summary, linked issue or product requirement when available, test evidence, and screenshots or recordings for UI changes. Call out any changes to the generation pipeline, output schema, pricing assumptions, or MVP scope.

## Security & Configuration Tips

Never commit API keys, model credentials, or private repository tokens. Store secrets in local environment files and document required variables in an example file such as `.env.example` once configuration is introduced.
